Lo que construimos

La Aplicación

Autenticación con roles
Registro de pacientes y login para 3 roles (Paciente, Médico, Admin) con JWT y control de acceso por endpoint.
Reserva de turnos
Especialidad → médico → fecha/hora → notas. Validación de fechas futuras y flujo de estados completo.
Panel del Médico
Agenda propia, confirmación de turnos, registro de completados y no-shows, disponibilidad semanal.
Panel de Administrador
Dashboard con métricas, gestión de médicos, especialidades, sedes, usuarios y log de auditoría.
Perfil de paciente
Edición de datos, historial de turnos con filtros, cancelación (ventana 24 h) y reprogramación.
API REST documentada
25+ endpoints en módulos, respuesta estándar JSON (success/data/timestamp/path), logging y auditoría.

Conceptos que defiendo en el examen

Diferencias clave

Caso de Uso vs Caso de Prueba
El caso de uso describe la interacción (qué hace el usuario). El caso de prueba la verifica con pasos y resultado esperado. El de prueba se deriva del de uso.
Carga vs Estrés
Carga = la carga esperada (¿soporta el uso normal/pico?). Estrés = excede la capacidad para hallar el punto de ruptura y ver si se recupera.
Severidad vs Prioridad
Severidad = impacto técnico del defecto. Prioridad = urgencia de corregirlo. Pueden no coincidir.
Funcional vs No Funcional
Funcional = ¿hace lo que debe? No funcional = ¿lo hace bien? (rápido, seguro, usable, confiable).

Retrospectiva

Aprendizajes del equipo

✓ Lo que funcionó
  • Postman cubrió toda la API y la seguridad
  • JMeter validó la concurrencia (0% error)
  • La revisión de código detectó el texto plano
  • Roles separados facilitaron el testing
⚠ Dificultades
  • JMeter sin backend → Connection refused
  • Estado del JSON mock entre corridas
  • Distinguir defecto real de mal precondicionado (DEF-04)
  • RNF-02 no estaba definido
💡 Aprendizajes
  • Definir criterios de entrada antes de ejecutar
  • Separar severidad de prioridad
  • La trazabilidad muestra los gaps, no solo lo OK
  • No reportar falsos positivos
